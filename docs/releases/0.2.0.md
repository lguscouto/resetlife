# ResetLife `0.2.0` — Persistência local das prioridades

**Data da validação:** 12/07/2026  
**Status:** build debug gerado e validado em emulador Android  
**Version code:** `2`  
**Version name:** `0.2.0`

## Resumo

A versão `0.2.0` substitui o armazenamento exclusivamente em memória por persistência local com Room/SQLite. As prioridades da tela Hoje são salvas por data, restauradas ao reabrir o aplicativo e mantêm o estado de conclusão.

## O que foi entregue

- entidade Room `PriorityEntity`;
- `PriorityDao` com observação por `dayKey`;
- banco `ResetLifeDatabase` versão 1;
- `PriorityRepository` com `Clock` injetável;
- persistência de inclusão e conclusão;
- filtragem de prioridades pelo dia atual;
- restauração das prioridades no `TodayViewModel`;
- estados de carregamento e erro de armazenamento;
- `ResetLifeApplication` e factory do ViewModel;
- exportação do schema Room;
- atualização de dependências e versão do aplicativo;
- cobertura unitária de domínio, repositório e ViewModel.

## Schema Room

O schema foi exportado pelo Room Gradle Plugin para:

```text
app/schemas/br.com.resetlife.data.local.ResetLifeDatabase/1.json
```

A configuração está em `app/build.gradle.kts`:

```kotlin
room {
    schemaDirectory("$projectDir/schemas")
}
```

O arquivo deve ser versionado para servir de histórico nas próximas migrações.

## Observações técnicas

A compilação ainda apresenta avisos não bloqueantes:

- KAPT faz fallback da linguagem Kotlin 2.0 para 1.9;
- Gradle informa funcionalidades depreciadas para versões futuras;
- uma biblioteca nativa do Compose não pôde ser stripada e foi empacotada sem strip.

Nenhum desses avisos impediu testes, geração dos APKs, instalação ou abertura do aplicativo.

## Limitações desta release

- não há sincronização em nuvem;
- não há login ou conta;
- não há testes instrumentados funcionais no source set `androidTest`;
- a validação manual confirmou instalação e abertura, mas não automatizou a interação completa de inserir/concluir uma prioridade;
- o banco está na versão 1; a primeira migração explícita será necessária quando o schema mudar.
